# Ethical AI Boundaries

This document expands on the [Ethics Policy](../ETHICS.md) for a technical audience. It explains what ethical constraints exist, how they are enforced in code and tests, and why they matter for anyone building AI-powered developer tools.

---

## What InterviewOps is built for

InterviewOps is a practice tool. It is intended to help developers:

- Run structured mock interview sessions before a real interview
- Identify knowledge gaps and weak areas using rubric-based scoring
- Practice behavioral storytelling using the STAR framework
- Understand what senior-level answers look like versus average ones
- Build a study plan based on their track and mode

The outcome the tool is optimizing for is **genuine skill development**, not the appearance of skill.

---

## What it explicitly refuses to build

The following features will never be added to InterviewOps:

| Prohibited feature | Why |
|-------------------|-----|
| Hidden screen overlays | Would allow running the tool invisibly during a live interview |
| Real-time answer injection | Would feed AI-generated answers to a candidate during an active session |
| Screen-share evasion techniques | Would help users hide the tool from interviewers |
| Stealth or background mode | Any mode designed to run undetected |
| Impersonation support | Generating content designed to misrepresent someone else's skills |
| Company-specific insider questions | Fabricating questions attributed to real companies' interview processes |

These are not missing features. They are intentional design exclusions.

## Technical boundaries

### No live session detection

InterviewOps does not detect whether a video call, screen share, or browser-based interview is active. This is an intentional omission. There is no code in the codebase that monitors network state, detects video conferencing apps, or responds to meeting software events.

### No screen capture

No component of InterviewOps reads from the screen, monitors clipboard contents, or captures screenshots. The CLI reads files from `input/` and writes files to `output/`. The Studio API reads and writes JSON files in `.interviewops-studio/`. Neither subsystem accesses screen or window state.

### No answer injection

There is no mechanism to pipe generated content into another application, overlay text on top of a browser or desktop window, or inject keystrokes into a focused window. Session output is written to Markdown files on disk only.

### Explicit ethics configuration

The Zod config schema in `src/config/config.schema.ts` encodes ethics constraints with locked defaults:

```typescript
export const EthicsConfigSchema = z.object({
  practice_only: z.boolean().default(true),
  block_live_cheating_features: z.boolean().default(true),
  include_ethics_notice: z.boolean().default(true),
  no_hidden_overlay: z.boolean().default(true),
  no_real_time_answer_injection: z.boolean().default(true),
});
```

These fields default to `true` and are validated at startup. They are checked in automated tests.

## The ethics-notice.md file

Every session generated by `simulate` includes an `ethics-notice.md` file in the session output directory. This file states:

- This output was generated for practice purposes only
- Prohibited uses: live interview use, real-time answer injection, hidden overlays, impersonation
- Permitted uses: pre-interview preparation, study, skill-building, feedback review

This file cannot be suppressed. The `verify` command checks for its presence, and a missing ethics notice causes verification to fail. The output writer will not complete a session without writing this file.

## How ethics are enforced in tests

`tests/ethics.test.ts` enforces several ethics properties automatically:

### Banned phrase test

A list of phrases that must never appear in session output is defined and tested against mock provider output:

```typescript
const BANNED_PHRASES = [
  'hidden overlay',
  'real-time cheating',
  'live answer injection',
  'screen share evasion',
  'stealth assistance',
  'bypass interview',
  'deceive interviewer',
];
```

If any of these phrases appear in generated content, the test fails. This prevents prompt injection or accidental wording that could encourage prohibited use.

### Ethics notice test

The test confirms that mock provider output contains the words `practice` and `ethics`, verifying the ethics notice is present in the generated content.

### Config defaults test

The test parses an empty config object and confirms that all ethics config fields default to `true`:

```typescript
expect(result.data.ethics.block_live_cheating_features).toBe(true);
expect(result.data.ethics.no_hidden_overlay).toBe(true);
expect(result.data.ethics.no_real_time_answer_injection).toBe(true);
expect(result.data.ethics.practice_only).toBe(true);
```

These tests run on every CI push and pull request against Node 20 and 22.

## Why this matters for AI engineers

If you are an AI engineer or building AI-powered developer tools, the ethics design decisions in InterviewOps are worth studying regardless of how you feel about interview practice tools specifically.

### Ethics by default, not by policy document

Having an ethics policy document is easy. Encoding ethical constraints into the type system, default configuration, and automated tests is harder and more durable. A policy document can be ignored. A failing test cannot be merged.

InterviewOps takes the position that ethics constraints should be:
1. Expressed as typed configuration with explicit defaults
2. Validated at startup, not assumed
3. Tested automatically in CI
4. Visible in every output artifact (ethics-notice.md)

### Trust is a design property

AI tools that can be repurposed for cheating, impersonation, or deception erode trust in the entire category of AI developer tools. Building an explicit, verifiable boundary between "practice" and "cheating" is not just ethics — it is product design that preserves the tool's value for honest users.

### Fairness has engineering implications

The ETHICS.md document states: "Other candidates prepare honestly. Using live AI assistance is unfair to them." For an AI engineer, this is a concrete design constraint: the system must not provide capability X, even if capability X is technically easy to build. Deciding what not to build is part of engineering.

## The ethics checklist in the PR template

The InterviewOps PR template includes an ethics checklist. Contributors must affirm:

- The change does not add live interview assistance
- The change does not add screen overlays or stealth features
- The change does not add real-time answer injection
- The change preserves `ethics-notice.md` in all session outputs
- The change does not claim this tool guarantees interview success

PRs that introduce prohibited features will not be merged.

## How contributors maintain these boundaries

1. **Read [ETHICS.md](../ETHICS.md)** before contributing — it is the canonical policy
2. **Run `npm test`** — the ethics tests run automatically; they cover config defaults and banned phrases
3. **Check the PR template** — the ethics checklist is there intentionally; fill it out honestly
4. **When uncertain, ask** — open a GitHub Discussion or issue before building a feature you are unsure about

The maintainer commitment in ETHICS.md is explicit: no live assistance features, no cheating workflows, ethics notice in all outputs, ethics policy visible in all documentation.
