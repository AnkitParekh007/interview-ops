# Coding Interview Rubric

## Scoring Scale
- **1 — Weak**: Significant gaps or inability to proceed without extensive help
- **2 — Below Expectations**: Partial solution with key issues unresolved
- **3 — Meets Expectations**: Working solution with some gaps
- **4 — Strong**: Clean, correct solution with good communication
- **5 — Exceptional**: Excellent solution, deep insight, exceptional clarity

---

## Dimensions

### 1. Problem Understanding (1–5)
Did the candidate understand the problem before starting? Did they clarify constraints?
- 1: Jumped straight in without reading the problem carefully; misunderstood key requirements
- 3: Understood the core problem; asked a few clarifying questions
- 5: Thoroughly clarified requirements, edge cases, and constraints before writing any code

### 2. Approach Quality (1–5)
Did they choose an appropriate algorithm or approach?
- 1: Chose an approach that fundamentally won't work or is O(n³) when O(n log n) exists
- 3: Workable approach, possibly not optimal
- 5: Identified optimal or near-optimal approach with clear rationale; considered alternatives

### 3. Code Correctness (1–5)
Does the solution actually work?
- 1: Code has fundamental bugs; doesn't run or produces wrong output
- 3: Mostly correct with minor bugs caught during review
- 5: Correct on first pass; handles all discussed cases accurately

### 4. Complexity Awareness (1–5)
Did the candidate reason about time and space complexity?
- 1: No mention of complexity; unaware of algorithmic trade-offs
- 3: Correctly stated time complexity; space complexity was skipped or wrong
- 5: Accurate analysis of both time and space complexity with clear reasoning

### 5. Edge Case Handling (1–5)
Did the candidate consider and handle edge cases?
- 1: No edge cases considered; solution breaks on null, empty, or boundary inputs
- 3: Caught most obvious edge cases after prompting
- 5: Proactively identified and handled edge cases before being asked

### 6. Testing (1–5)
Did the candidate test or trace through their solution?
- 1: No testing; didn't trace through example inputs
- 3: Traced through provided example; no additional test cases
- 5: Created multiple test cases including edge cases; wrote or discussed unit tests

### 7. Communication (1–5)
Did the candidate think aloud and communicate their process clearly?
- 1: Silent coding; interviewer couldn't follow thought process
- 3: Occasional narration but significant stretches of silence
- 5: Continuous, clear narration of thought process; engaged the interviewer as a collaborator

### 8. Seniority Signal (1–5)
Does the quality of the solution match the level being hired for?
- 1: Junior-level solution for a senior role
- 3: Matches level with minor gaps
- 5: Demonstrates clear expertise for the seniority level; teaches the interviewer something
